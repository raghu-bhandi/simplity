Record

Table, entity and data-structure are the other possible names for this component. Essentially we define a group of fields into a record and use it for different purposes. A record helps us in implementing one of the golden rules of design : DRY (Do not Repeat Yourself) Once a record is defined, we can use to represent a database table/view, or a data structure for a parameter for a stored procedure, or set of fields that are expected as input from client etc. We have three types of records:

  • Storage: represents a record that is stored as part of your data base. A storage record maps directly to a table in your RDBMS. Note that only storage records can be used for writing to the data base. (add/modify/delete)
  • View: represents a record as seen by end-users. Typically a view has either a subset of fields from a storage record, or it is formed by picking fields from a set of related storage records. For example when you show details of a customer on a page, you may want to show total order value of this customer, number of pending orders etc.
  • Structure: this is a set of related fields that are utilized by some part of your application, but they are not directly mapped to any field in a storage record. These fields are temporary data elements that are calculated at run time implement some business rules.

Example

<?xml version="1.0" encoding="UTF-8"?>
<record xmlns="http://www.simplity.org/schema" keyToBeGenerated="false" name="Employees" tableName="EMPLOYEES" moduleName="birt" okToSelectAll="true" suggestionKeyName="firstName" suggestionOutputNames="firstName" >
<fields>
<field name="employeeNumber" dataType="id" fieldType="primaryKey" columnName="employeeNumber" label="Id" />
<field name="lastName" dataType="text" columnName="lastName" label="lastName" />
<field name="firstName" dataType="text" columnName="firstName" label="firstName" />
<field name="extension" dataType="text" columnName="extension" label="extension" />
<field name="email" dataType="text" columnName="email" label="email" />
<field name="officeCode" dataType="text" columnName="officeCode" label="officeCode" />
<field name="reportsTo" dataType="id" columnName="reportsTo" label="reportsTo" />
<field name="jobTitle" dataType="text" columnName="jobTitle" label="jobTitle" />
</fields>
</record>

Attributes

Attribute Required? Description
Name yes Name of this record/entity, as used in application
moduleName no Module name + name would be unique for a component type within an application
recordType no
type of this record
Record Type: storage, view, structure
tableName no name of the rdbms table, if this is either a storage table, or a view that is to be defined in the rdbms
keyToBeGenerated no Has this table got an internal key, and do you want it to be managed automatically?
okToSelectAll no If this table is expected to have large number of rows, we would like to protect against a select with no where conditions. Of course one can always argue that this is no protection, as someone can easily put a condition like 1 = 1
childrenToBeRead no Child records that are to be read whenever a row from this record is read.
childrenToBeSaved no Child records to be saved along with this record. operations for this record
Fields no fields that this record is made-up of
valueListFieldName no In case this is a table that supplies key-value list for drop-downs, then we use primary key as internal key. Specify the field to be used as display value. key of this table is the internal value
valueListKeyName no Relevant only if valueListFieldName is used. If the list of values need be further filtered with a key, like country code for list of state, specify the that field name.
defaultSheetName no What is the sheet name to be used as input/output sheet. (specifically used in creating services on the fly)
keyGeneratorName no If keys are to be generated, we may have a sequence, as in oracle. Null implies that RDBMS auto-increments and we are not to set any value to this column
suggestionKeyName no if this record is used for a suggestion service, field that is used for search
suggestionOutputNames no what fields do we respond back with for a suggestion service
readOnly no Is this record only for reading?
schemaName no If this application uses multiple schemas, and the underlying table of this record belongs to a schema other than the default, then specify it here, so that the on-the-fly services based on this record can use the right schema.
